# Logistics Integration Plan — 店到店物流實用化路線

更新日期：2026-06-26

## 主管結論

如果二手安心交易要從「流程紀錄工具」變成真正實用的二手交易工具，物流一定要接。原因很簡單：買賣雙方最常卡住的不只是付款，而是「怎麼寄、運費多少、包裹能不能寄、物流進度到哪、買家不取貨怎麼辦」。

但第一階段不要直接重串 7-11、全家、萊爾富官方物流。正式取號、列印標籤、逆物流、貨態 webhook 都會牽涉合約、個資、客服與責任邊界。MVP 應該先做「物流決策輔助層」，再逐步走向正式 API 串接。

## 我們要解決的物流痛點

1. 賣家不知道包裹尺寸/重量能不能寄。
2. 買方不知道運費誰付、多少錢、多久到。
3. 雙方不知道物流進度，容易一直私訊追問。
4. 買家不取貨時，賣家不知道怎麼留下紀錄。
5. 商品損壞或遺失時，缺少包裝照片、物流單、寄件時間線。

## 三階段整合策略

### Phase 1：物流規格與運費試算，不取號

這階段不碰物流 API，不處理個資，不幫使用者建立正式物流訂單。

功能：

- 選擇物流：7-11、全家、萊爾富。
- 顯示尺寸/重量限制。
- 顯示預估運費欄位，可先手動設定。
- 建議包裝留證：包裝前照片、封箱照、寄件單照。
- 交易頁新增物流狀態：待寄件、已寄件、運送中、已到店、已取件、退回中。
- 賣家可輸入物流單號或寄件代碼，先做紀錄，不查官方貨態。

好處：

- 不需要合約。
- 不碰物流個資 API。
- 可以立刻改善買賣雙方溝通成本。

### Phase 2：外部物流頁導流與手動貨態

這階段仍不自動取號，但讓賣家更快完成寄件。

功能：

- 產生「請到 7-11 / 全家 / 萊爾富建立寄件單」的操作指引。
- 交易頁提供欄位讓賣家貼上物流單號。
- 買方頁顯示「賣家已填物流單號，請自行至物流商查詢」。
- 物流進度由賣家/買方手動更新，作為交易證據時間線。

### Phase 3：第三方物流 API 正式串接

等有使用量後，再評估串接綠界、ezShip 或直接談超商物流合作。

正式 API 目標：

- 門市選擇。
- 建立物流訂單。
- 取得寄件編號。
- 列印寄件標籤。
- 貨態查詢或 webhook。
- 退貨/逆物流流程。

## 為什麼優先考慮第三方物流 aggregator

直接跟 7-11、全家、萊爾富逐一談，對小團隊太重。第三方物流 API 通常已經整合多家超商，可以降低串接與商務成本。

可評估：

- 綠界物流 API：支援全家、7-ELEVEN、萊爾富等 C2C/B2C 超商取貨，也提供物流選擇頁面與暫存物流訂單機制。
- ezShip：可研究是否符合個人賣家、小量出貨情境。
- 各超商自家賣場工具：適合導流，不一定適合直接 API 串接。

## 我們產品上的物流資料模型

交易建立時新增：

- `shipping_provider`：seven_eleven / familymart / hilife / manual
- `shipping_fee_payer`：buyer / seller / split
- `shipping_fee_estimate`：預估運費
- `package_size`：S / M / L / custom
- `package_weight_kg`
- `tracking_no`：物流單號或寄件代碼
- `ship_from_store`：寄件門市名稱，可選填
- `ship_to_store`：取件門市名稱，可選填
- `shipping_status`：pending / shipped / in_transit / arrived / picked_up / returned / exception

## 頁面設計建議

### 建立交易頁

新增「物流設定」區塊：

1. 物流方式：7-11 / 全家 / 萊爾富 / 面交 / 其他。
2. 包裹尺寸：長、寬、高、重量。
3. 系統顯示：是否可能超過店到店限制。
4. 運費：賣家填預估運費，或系統提供常用值。
5. 運費誰付：買方 / 賣方 / 分攤。

### 交易狀態頁

新增「物流時間線」：

- 待寄件。
- 賣家已寄件。
- 已填物流單號。
- 買方確認到店通知。
- 買方已取件。
- 未取退回。

## 法務與責任邊界

在未正式串接 API 前，必須明確標示：

- 本服務不代為寄件。
- 本服務不保證物流時效。
- 本服務不承擔物流遺失、破損、延誤賠償。
- 實際運費、尺寸、重量、禁運品以物流商公告為準。
- 使用者不得寄送違禁品、易碎品、生鮮、活體、現金、有價證券等高風險品項。

## MVP 優先順序

1. 先做物流方式與尺寸/重量提醒。
2. 再做運費欄位與誰負擔運費。
3. 再做物流單號欄位與手動貨態事件。
4. 等真實使用量出現，再串物流 API。

## 不要現在做

- 不要直接承諾一鍵取號。
- 不要自己保管買賣方物流個資。
- 不要做物流理賠仲裁。
- 不要同時串三家官方 API。
- 不要把物流整合包裝成我們已經正式合作。

## 下一步 sprint 建議

### Sprint 1：物流資訊層

- 首頁新增物流能力說明。
- 建立交易表單新增物流方式、運費、尺寸重量欄位。
- 交易頁顯示物流條件。
- 不串 API。

### Sprint 2：手動物流進度層

- 允許賣家填物流單號。
- 允許買方/賣方新增物流事件。
- Dashboard 統計物流方式使用比例。

### Sprint 3：API 評估層

- 測綠界物流 API sandbox。
- 確認取號、標籤、貨態查詢、退貨流程。
- 確認個資保存與刪除策略。
